# План разработки — путь к первым деньгам (КалориСкан)

## Контекст

Продукт технически готов: анализ фото (Claude Opus vision), ручной ввод блюда, сканер штрихкодов, дневник питания/вода/вес/цели — всё на localStorage, без аккаунтов и без бэкенд-хранилища пользовательских данных. Фото перед отправкой на `/analyze` теперь ресайзятся на клиенте (снижена себестоимость скана). Маркетинговый план и SEO-лендинг существуют, но ещё не запущены. Юридический статус для приёма платежей (самозанятый/ИП) пока не подтверждён.

**Фокус:** окупаемость строим только на платежах пользователей (freemium-подписка). Реклама сознательно исключена из этого плана — это тема отдельного, более позднего этапа развития, не текущего.

**Цель плана:** не полноценная подписочная платформа, а минимальный набор шагов, чтобы получить первый реальный платёж и проверить готовность людей платить — прежде чем вкладываться в аккаунты, БД, рекуррентный биллинг.

**Ключевое решение:** concierge/manual MVP — ручная оплата + ручная выдача доступа, вместо полной автоматической подписки с вебхуками. На порядок быстрее в реализации и достаточно для проверки спроса на первых платящих.

**Обновлено 2026-07-07 — старт без регистрации:** первые **17 платежей** принимаются напрямую по **СБП** (личная ссылка на перевод по номеру телефона через банковское приложение — Т-Банк/Сбер и т.п.), без самозанятости и без ЮKassa. Это зафиксировано в коде — `/admin/grant` физически откажет в выдаче доступа после 17-го платежа (`db.MAX_PAYMENTS_BEFORE_REGISTRATION`), а пейволл покажет «приём приостановлен» вместо кнопки оплаты. После 17-го платежа — **остановка приёма и обязательная регистрация самозанятости**, дальше уже можно подключать ЮKassa для автоматизации. Причина: системный доход на личную карту без регистрации рискует блокировкой счёта по 115-ФЗ (банки отслеживают регулярные переводы от разных отправителей с 1 июня 2025) — на 17 разовых платежах риск невелик, но не более того.

**Цена подписки: 190 ₽/месяц** (зафиксировано в `app.js`, `SUBSCRIPTION_PRICE_RUB`) — обновлено 2026-07-07, ближе к расчётной цене для 40% рентабельности (см. ниже).

## Верификация (после реализации приоритета 0)

Ручной прогон полного пути: исчерпать лимит сканов на тестовом `device_id` → увидеть пейволл → перейти по ссылке оплаты → (тестовый/реальный платёж) → вручную выдать pro через admin-endpoint → убедиться, что лимит снят для этого `device_id`. Проверить, что события (`paywall_shown`, `buy_click`, `payment_granted`) пишутся в таблицу `events`.

Кроме happy path, проверить и негативные сценарии:
- повторная оплата уже pro-пользователем — не должна дублировать/ломать `pro_until`
- восстановление доступа по email после потери `device_id` (переустановка PWA, смена устройства, очистка данных)
- поведение UI при обрыве соединения в момент перехода на оплату
